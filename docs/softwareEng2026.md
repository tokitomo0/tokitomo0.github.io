## ソフトウェア工学の本質と必要性
### ソフトウェア工学の定義
* 実務から体系化された知見。知見の理解と実践には乖離があり、チーム全体での共有が必要。
* IEEEによる定義：ソフトウェアの開発、運用、メンテナンスに対するシステマティックで規律（discipline）ある、定量化できるアプローチの適用、すなわちソフトウェアに対するエンジニアリングの適用。
* ソフトウェア工学のゴール：進化論に基づく（ソフトウェアシステムが変化し続ける状況に対応する）方法論を考え出すことである。新しいソフトウェアが古いシステムから構築され、すべてが互換性を持ち、他システムと相互運用しなければならない。

### ソフトウェアの特徴
* ハードウェアと異なり劣化しないが、悪化する。
* 新しい環境や技術、ニーズを満たすよう適応しなければならない。
* 新しいビジネス要求を実現するために強化しなければならない。
* より近代的なシステムやデータベースと相互運用するため拡張しなければならない。
* 進化し続ける環境で実行可能となるよう、再構築しなければならない。

### プログラミングとソフトウェア工学の違い
* **時間**
    * プログラムの想定稼働期間、依存するライブラリ・OS・HWへの対処、持続可能性の検討。
* **スケール**
    * 開発・保守作業のスケーラブル化、対応する人員のスケーリング。
* **時間・スケールのトレードオフ**
    * 両者の最適化。

### Hyrumの法則
* あるAPIに十分な数のユーザーがいるとき、APIを作った者自身が契約仕様として何を約束しているかは重要ではない。作られたシステムが持つあらゆる観察可能な（observable）な挙動に関して、それに依存するユーザーが出てくるものである。

### スケーラブルなソフトウェア
* 変更すべきすべてのものを安全に変更可能で、かつコードベースの存続期間を通してそれが可能であるとき、組織のコードベースは持続可能である。
* スケールする必要がある有限なリソース：人件費、コンピュータ、メモリー、ストレージ、ソフトウェア開発者の時間的関与と開発ワークフロー。
* コストの種類：財務コスト、資源コスト（CPU時間等）、人件コスト、処理コスト、機会コスト、社会コスト。

### Hookerの7原則
1. システムが存在する唯一の理由：システムはユーザーに価値を提供するためにある。
2. シンプルにしておけ（KISS: Keep It Simple, Stupid）：すべての設計は可能な限りシンプルであるべき。
3. ビジョンを持ち続けよ：ソフトウェアプロジェクトの成功に明確なビジョンが不可欠である。
4. あなたが作ったものを他の人が使用する：他人が理解しなければならないことを意識して仕様・設計・ドキュメント・実装を決める。
5. 未来へオープンであれ：環境や仕様変更に対応すべく、一般的な問題を解決するシステムを構築し選択肢を与える。
6. 再利用に先駆けて計画せよ：コストを低減し、再利用するコンポーネントとそれを取り込むシステムの両方の価値を高める。
7. 考えよ：行動を起こす前に簡潔明瞭な考えをもつことは、必ず良い結果を生む。

### ソフトウェア工学の必要性
1. **大規模化と複雑さ**：ソフトウェアなしで社会が回らないため、効率よい開発・メンテナンスと効率・品質の両立が必要。部品化、オブジェクト化が有効。
2. **IT人材の不足**：需要に対する量的不足と、効率的開発・管理ができる人材（課題解決型＋価値創造型、スペシャリスト、PM人材）の質的不足。
3. **社会性と求められるミッション**：ライフライン・インフラとしての増加。不測の事態における事業・サービス継続の要求。
4. **不確実性の増大**：利用の変化・多様化（検索型から交流型へ、意図しない活用など）。

### 現代の課題・事例
* **未経験者の参入**：非情報学分野からITエンジニアへの転身における、知識と実務イメージのギャップ。
* **システム更改対応**：セキュリティパッチやバグ対応など、ハードウェア・ソフトウェアの定期的な改良の継続。
* **FOSS4G活動**：プロのプログラマによるボランティアとしてのソフトウェア開発実態。
* **Individual Contributor**：Rパッケージ開発等におけるテストや他パッケージ依存問題。
* **AI・機械学習**：Deep Learningモデルのパラメータ調整。AI Agentによるソフトウェア開発における要件定義、バージョン管理、テストの不可欠性。人間には監督者としてのソフトウェア工学の能力が求められる。

## ソフトウェアライフサイクル
* **誕生（ニーズの発生）**：ビジネス・システム上の要求を具現化する企画・計画、要件定義書の作成。
* **開発・運用**：開発、実稼働、保守運用。
* **廃止**：サービスの終了、新規ソフトウェアへのリニューアル。

### 要件定義
* 項目：システムの目的、概要、機能、システム構成、目標性能、他システムとのインターフェイス仕様、運用面の注意事項、制限事項、拡張性、開発スケジュール、開発体制、納品物。
* 実現する機能・実現しない機能を明確にすること（Feasibility Studyによる実現性評価、機能絞り込み、代替策検討）。検討漏れを防ぐためのテンプレート化。
* 意義：
    * プロジェクト関係者のスコープ共有（バイブル）。
    * 社内外へのプロジェクト内容明文化と協力依頼。
    * RFP（Request for Proposal：外部委託の際、ベンダーへシステム提案を要求する資料）の作成。

### 設計
* 要件定義書から、コーディングができるレベルの仕様書（クラス、モジュール等）への落とし込み。WBSの策定。

### 制作（調達）
* 内製（自社開発）または調達（外部委託）。
* 外部委託時、ソースコードを納品物に含める場合は自社メンテが可能。含めない場合はメンテナンス作業委託契約が必要。
* オフショア開発：賃金の安い海外企業への外部委託。文化・言語・商習慣・品質意識の違いに留意。将来的に日本がオフショア開発される側になる可能性。

### テストとデバッグ
* バグのないソフトウェアはない前提に立つ。早期発見・対処が理想だが、工数限界のためすべてをテストすることは不可能。どの箇所をどの程度テストするかの選定が必要。

### 運用・保守
* 稼働後のデータをプロアクティブに活用すべき。ソフトウェア以外の要因も考慮が必要。開発者がプロジェクトから離れている可能性を考慮する。

### ドキュメンテーション
* 複数人開発における非同期コミュニケーション（文書による意思決定・伝達）の重視。
* 海外プロジェクト増加やIT技術・エラーメッセージの性質上、基本的に英語ベースとなる。

## プロジェクト管理とソフトウェア分析
### プロジェクトの特性
* **有期性**：必ず終わりがある（ルーティンワークではない）。
* **独自性**：独自の目的を達成する（例：ソフトウェア開発）。

### 計画のアプローチ
* **フォアキャスティング**：インプット視点（どれだけやったか）。現在の立ち位置から順に目標へ進む。目標への未到達や、意図しない場所へ到達するリスクがある。
* **バックキャスティング**：アウトプット視点（何を成し遂げたか）。定めた目標（イシュー）に向けて必要な計画を逆算する。目標や筋道は適宜修正可能。プロジェクト業務はバックキャスティングである。

### ソフトウェア分析と評価
ソフトウェアを見える化し、QCD（Quality, Cost, Delivery）の最適バランスを実現するための手法。
* 優先度：品質優先（人命に関わるシステム）、費用優先（公共関連システム）、納期優先（イベント行事関連システム）。

#### 評価指標
1. **コード物量（ステップ数）**：C言語換算などのソースコード行数（総ステップ数）。複数言語混在時は換算が必要。
2. **コード物量（オブジェクト容量）**：組み込み型ソフトウェアなどではサイズが小さいほど良い。
3. **ファンクションポイント（FP）法**：プログラミング前にユーザー要件からシステム規模を概算する手法。投資意思決定や新規開発・購入の比較指標。
    * FP値 = 基準値 * (0.65 + 調整値 / 100)
    * 原価 = FP値 * 作業単価 * 換算値
4. **使い勝手**：画面の視認性、操作性、入力補助、互換性、ガイダンス。

#### 品質管理とバグ発生率
* 品質証明：「計画していたテストで全バグに対応した」「まだバグが確認されていない」状況でもHyrumの法則を想定すべき。リリース時期の判断（バグ最小化か、リリース後対応か）にも関わる。
* **バグ発生率** = 発生バグ数 / 1ステップ数（改行を除くコード行数）
    * 例：100ステップに100テスト実施、10件バグ発生 -> バグ発生率1 = 10 / 100 = 0.1 件/ステップ。
    * 追加で50テスト実施、5件バグ発生 -> バグ発生率2 = 15 / 100 = 0.15 件/ステップ。
    * 追加テストにより数値が上がる性質を持つため、一概に品質悪化とは言えず、評価の1指標として扱う。

### WBS（Work Breakdown Structure）
* プロジェクト目標達成・要素成果物生成のために、作業を階層的に要素分解した構成図。作業を細かく分解し、作業順に並べる。
* メリット：スコープ明確化（やらないことの明確化）、作業の洗い出し、全体管理・作業計画の明確化。
* 作り方：
    * バックキャスティング、トップダウンアプローチによるスコープ明確化。
    * 大きな作業のグルーピング。
    * グルーピングした作業の相互関連性の整理（順序関係）。
    * 各グループの作業を、MECE（Mutually Exclusive and Collectively Exhaustive：モれなくダブりなく）に洗い出す。

## 開発プロセス
### ウォーターフォール型開発プロセス
* 進捗管理が容易で成果物が明確。後工程にしわ寄せが集中するリスクがある。

### スパイラルモデル
* 開発を小さなフェーズに分割し、フェーズごとにプロトタイプによるデモとフィードバックを行う。プロトタイプ作成に想定外の作業量が発生するリスクがある。

### 反復型開発プロセス
* ソフトウェアを機能分割し、「反復」単位で管理する積み上げ方式。
* メリット：顧客要求を取り入れやすい、部分的な納品が可能。
* デメリット：分割のための管理業務の増加、全体像の見えにくさ、一括稼働システムでは分割の意味がなくなる点。

### アジャイルプロセス
* 変化に対応して無駄を廃し、最適な手法で動くソフトウェアの提供を優先する。「できるだけ決定を遅らせる」「できるだけ早く提供する」を実現する。
* **アジャイルの価値**：
    * プロセスやツールよりも、個人や相互作用。
    * わかりやすいドキュメントよりも、動くソフトウェア。
    * 契約上の駆け引きよりも、顧客とのコラボレーション。
    * 計画を硬直的に守るよりも、変化への対応。
* **コンセプト・キーワード**：変化の受容性、短いサイクル／頻繁な納品、シンプルな設計、リファクタリング、ペアプログラミング、レトロスペクティブ（振り返り）、暗黙知、テスト駆動型開発、スクラム、カンバン、XP（eXtreme Programming）。

#### アジャイル開発の12原則
1. 早期・継続的な納品による顧客満足の最優先。
2. 開発終盤の要求変更の歓迎（変化を味方につける）。
3. 数週間〜数ヶ月の短いサイクルでの納品。
4. ビジネス側の人人間と開発者の日々の協力。
5. 意欲ある個人を中心としたプロジェクト編成と環境・支援・信頼の提供。
6. 最も効率的な情報伝達手段としての対面での会話。
7. 動くソフトウェアによる進捗測定。
8. 持続可能な開発ペースの維持。
9. 技術的卓越性と優れた設計への継続的な注意。
10. 簡潔さ（無駄を省くこと）の本質性。
11. 自己組織化されたチームからのアーキテクチャ・要求・設計の創出。
12. 定期的なチームの振り返りとプロセスの調整。

#### 特徴的な概念
* **YAGNI**：You Aren't Going to Need It.（必要ないものは作らない）
* **CRACK**：顧客の理想的な状態。Collaborative（協力的）、Representative（意思がはっきりしている）、Authorized（権限がある）、Committed（献身的）、Knowledgeable（知識がある）。

### 手法の選定とリスク駆動型
* 開発手法に絶対的な正解はなく、場面や要求、PMスキルに応じてハイブリッド型も含めて選定する。
* **リスク駆動型開発プロセス**：想定されるリスクを予め把握し対処する。
    * アジャイル：人的リスク。
    * 計画駆動型：技術的課題、規模の拡大／縮小。
    * 共通：ニーズの変化、政治的要因、社会的要因。
    * 予測不可能：天災、メンバーの事故・病気、開発環境のクラッシュ。

## コーディングとスタイルガイド
### コーディングの基本原則
* **読みやすさの重視**：コードは書かれることよりも読まれることの方が多い。他人はもちろん、未来の自分が読みやすいコードを書くために、共通のルール（スタイルガイド）が必要である。
* **一貫性の重要性**：スタイルガイドに合わせることも重要だが、プロジェクト全体で一貫性を保つことが最も重要である。

### Python PEP 8 の主要なルール
* **1行の長さ**：コードの1行は79文字以内にする。docstringやコメントは72文字以内にする。
* **インデント**：1レベルのインデントごとにスペースを4つ使用する（タブは使用しない）。
* **行の継続**：折り返された要素は、縦に揃える（Hanging indentation）。
* **スペースの入れ方**：
    * 演算子の前後にスペースを1つずつ空ける。ただし、優先度の異なる演算が混在する場合は、高優先度の演算子の前後にスペースを入れない（または調整する）ことで優先度を視覚的に表現する。
    * 無駄なスペース（括弧のすぐ内側など）は入れない。
* **改行**：複数の文を1行に重ねて書かない。
* **インポート（import）の順序**：以下の順にグループ分けし、それぞれ1行空けて記述する。
    1. 標準ライブラリ
    2. サードパーティに関連するもの
    3. ローカルなアプリケーション/ライブラリ特有のもの
* **エンコーディング**：UTF-8以外は可能な限り使わず、ASCII文字以外の使用（日本語のコメントなど）を極力避ける。

### 命名規則（Naming Conventions）
* **パッケージ**：すべて小文字の短い名前。アンダースコアは使わない。
* **モジュール**：すべて小文字の短い名前。必要に応じてアンダースコアで区切る。
* **クラス、例外**：CapWords方式（先頭のみ大文字の単語を繋げる。アンダースコアは使わない）。
* **関数、メソッド**：小文字のみ。必要に応じて単語をアンダースコアで区切る。
* **定数**：すべて大文字。単語をアンダースコアで区切る。通常、モジュールレベル（関数の外側）に定義する。
* **変数、引数**：小文字のみ。必要に応じて単語をアンダースコアで区切る。
* **避けるべき1文字変数**：`l`（小文字のエル）、`O`（大文字のオー）、`I`（大文字のアイ）は、フォントによって数字の 1 や 0 と見分けがつかないため決して使用しない。

### 静的解析ツール（Linter）
* コードがスタイルガイドに従っているかを自動でチェックするツール。
* **flake8**：PEP 8に基づく代表的なコードチェックツール。
* **pylint**：VS Codeのデフォルトなどに採用されている強力な解析ツール。
* **エラーコードの分類**：
    * **F**：コードのエラー（Code error）
    * **E**：スタイルのエラー（Style error）
    * **W**：スタイルの警告（Style warning）

## バージョン管理とGit
### バージョン管理システム（VCS）
* ファイルを「誰が」「いつ」「どのように」変更したかを記録・管理し、複数人での共同開発を可能にするシステム。
* **集中管理型（Subversionなど）**：
    * 特徴：リモートリポジトリのみでバージョンを一元管理する。
    * メリット：構造がシンプルで理解しやすい。
    * デメリット：同時編集によるコンフリクトが発生しやすく、サーバー障害時に作業が停止する。
* **分散管理型（Gitなど）**：
    * 特徴：各開発者が手元に「ローカルリポジトリ」を持ち、履歴を分散して保持する。
    * メリット：ローカルで高速に履歴管理ができ、障害に頑健。リモートへのアクセス頻度が低い。
    * デメリット：概念の理解や操作がやや複雑。

### Gitの基本概念
* **コミット（Commit）**：ファイルの作成・変更・削除の「記録」。メッセージ（「どのような変更か」）を添えて、任意のタイミング・単位で保存する。原則として過去のコミットは変更・削除できず、打ち消す際もその履歴が残る。
* **リポジトリ（Repository）**：Gitが管理するプロジェクトフォルダ。
    * **ローカルリポジトリ**：開発者個人の実行環境。
    * **リモートリポジトリ**：GitHub等に配置される共有の管理場所。

### ローカル開発の流れとファイルの3状態
ローカル環境では、ファイルは3つのエリアを遷移する。
`[ ワークツリー ] -> ( git add ) -> [ ステージングエリア ] -> ( git commit ) -> [ Gitディレクトリ ]`

* **ワークツリー（Worktree）**：実際にファイルを編集する作業スペース。
    * **untracked**：Gitの管理対象外のファイル。
    * **unmodified**：管理対象だが、前回から変更がない状態。
    * **modified**：管理対象で、変更が加えられた状態。
* **ステージングエリア（Staging Area / Index）**：コミットする対象ファイルを登録する一時領域。
    * **staged**：`git add` によって登録された状態。
* **Gitディレクトリ（Git Directory）**：`git commit` によって変更履歴が永久に記録される場所。コミットされたファイルは再び `unmodified` 状態に戻る。

### 主要なGitコマンド
* **設定・確認系**：
    * `git init`：ローカルリポジトリの新規作成・初期化。
    * `git status`：ワークツリーとステージングエリアの状態を表示。
    * `git config`：ユーザー名やメールアドレスなどの設定確認・変更。
    * `git log`：コミット履歴の表示（`--oneline` で1行表示）。
    * `git diff`：ファイル変更の差分を表示。
* **コミット・修正系**：
    * `git add`：ファイルをステージングエリアに追加。
    * `git commit`：ステージングされた変更を記録。
    * `git commit --amend`：直前のコミットを修正。
    * `git checkout`：過去のコミットの復元、または作業ツリーの変更取り消し。
    * `git reset`：特定のコミットまで履歴を巻き戻す（ローカル用）。
    * `git revert`：過去のコミットを「打ち消す」新たなコミットを作成する（安全な履歴修正）。
    * `git rm`：ファイル自体の削除およびインデックスからの削除。
* **リモート・ブランチ連携系**：
    * `git clone`：リモートリポジトリをローカルに複製。
    * `git push`：ローカルの変更をリモートリポジトリにアップロード。
    * `git pull`：リモートリポジトリの最新変更をローカルに取得して同期。
    * `git branch`：ブランチの作成・一覧表示。
    * `git merge`：異なるブランチの変更履歴を統合。
    * `.gitignore`：Gitの管理対象外（パスワード、一時ファイル、OS依存ファイル等）にするファイルを指定する設定ファイル。

## GitHubと共同開発手法
### GitHubとは
* Gitのリモートリポジトリをホスティングするサービス。オープンソースソフトウェア（OSS）の開発・共有における主要なプラットフォーム。

### 協調開発の機能
* **Fork（フォーク）**：他人のリモートリポジトリを、自分のGitHubアカウント配下に複製する。
* **Clone（クローン）**：リモートリポジトリをローカルマシンにダウンロードする。
* **Pull Request（プルリクエスト）**：自分が加えた変更を、オリジナルのリポジトリ（または親ブランチ）に取り込んでもらうよう提案・依頼する機能。
* **Issues（イシュー）**：バグ、タスク、機能追加、質問などをチケットとして起票し、議論・管理する機能。
* **Projects（プロジェクト）**：イシューをカンバン方式やテーブル方式で視覚的に進捗管理する機能。

### Git / GitHubフロー
* **Git Flow**：`master`、`develop`、`feature`、`release`、`hotfix` など、複数の役割を持つブランチを厳密に使い分ける大規模開発向けの手法。
* **GitHub Flow**：`main` ブランチと、作業用の使い捨てブランチのみを用いるシンプルな開発モデル。常に `main` はデプロイ可能（クリーン）な状態を維持し、すべての変更はプルリクエストを経由してマージされる。

## CI/CD（継続的インテグレーション／継続的デリバリー）
### CI/CDの定義
* **CI（Continuous Integration：継続的インテグレーション）**：
    * 開発者がコード変更を頻繁に共有リポジトリに統合するプロセス。コードのプッシュやプルリクエストをトリガーに、自動ビルドと自動テストを実行することで、バグを早期に発見・修正する。
* **CD（Continuous Delivery：継続的デリバリー）**：
    * テスト済みのコードをステージング環境や本番環境へ自動的にデプロイするプロセス。手動デプロイによる人的ミスを排除し、迅速に価値（製品）をユーザーへ提供可能にする。

### CI/CDパイプラインのステップ
1. **ソース**：コード変更が検知され、ワークフローが起動。
2. **ビルド**：ソースコードのコンパイルや環境構築。
3. **テスト**：静的解析（Linter）やユニットテスト、統合テストの自動実行。
4. **デプロイ**：テストをパスした安全なコードを環境へ適用。
5. **検証**：本番・テスト環境での基本的な動作・接続確認。
6. **モニタリング**：サービス稼働後の継続的な監視。

### GitHub Actionsによる自動化
* GitHub上で直接ビルド、テスト、デプロイのパイプライン（ワークフロー）を構築できるサービス。
* **基本構造と概念**：
    * **ワークフロー（Workflow）**：実行する自動プロセス全体。YAML形式（`.github/workflows/*.yml`）で定義する。
    * **イベント（Event）**：ワークフローを起動するトリガー（例：`push`、`pull_request`）。
    * **ジョブ（Job）**：同じ実行環境（Runner）で実行されるステップの集まり。デフォルトでは並列で実行される。
    * **ステップ（Step）**：ジョブ内で順番に実行される個々のタスク。コマンド（`run`）の実行や、再利用可能な「アクション」の呼び出しを行う。
    * **アクション（Action）**：よく使われる定型処理をパッケージ化したもの。コミュニティが公開しているアクション（例：`actions/checkout`、`actions/setup-python`）を呼び出して利用できる。
    * **ランナー（Runner）**：ワークフローが実行される仮想マシン（GitHub HostedまたはSelf-Hosted）。

## ドキュメンテーションとGitHub Pages
### GitHub PagesとJekyllの活用
* **GitHub Pages**：GitHubリポジトリ内のファイルをWebページとして直接公開できる静的サイトホスティングサービス。
* **Jekyll（ジキル）**：Markdownファイルを自動的にHTMLに変換して静的サイトを生成するツール。GitHub Pagesと密接に統合されており、開発者はMarkdownで文書を書くだけでドキュメントサイトを構築・公開できる。
* **CI/CDとの融合**：ドキュメントのソース（Markdown等）をGitで管理し、リポジトリへのプッシュをトリガーにGitHub Actionsを通じて最新のWebサイトへ自動ビルド・デプロイを行う。
